# Design Decisions

This note captures the key trade-offs made in egras — the *why* behind choices that might otherwise seem arbitrary. Understanding these helps you make consistent decisions when extending the codebase.

---

## Atomic Registration

**Decision:** User creation and org membership are a single database transaction, not two separate calls.

**Context:** Registering a user requires two writes: `INSERT INTO users` and `INSERT INTO user_organisation_roles`. If the first succeeds and the second fails, the user exists but belongs to no organisation — a "zombie" user that cannot log in.

**Choice:** `UserRepository::create_and_add_to_org` wraps both inserts in a `BEGIN ... COMMIT` block. If the role assignment fails (e.g., bad role code, org doesn't exist), the user insert is rolled back.

**Trade-off:** Slightly more complex repository implementation, but eliminates an entire class of data inconsistency.

See [`src/security/persistence/user_repository_pg.rs`](../../src/security/persistence/user_repository_pg.rs).

---

## 404 vs 403 for Cross-Org Access

**Decision:** When a caller tries to access a resource in an organisation they're not authorised for, return **404 Not Found**, not 403 Forbidden.

**Context:** If accessing org `B` as a member of org `A` returned 403, the attacker learns that org `B` exists. Returning 404 makes the response indistinguishable from a genuinely non-existent organisation — it hides the tenant's existence entirely.

**Choice:** `authorise_org()` in [`src/auth/permissions.rs`](../../src/auth/permissions.rs) returns `AppError::NotFound { resource: "organisation" }` on cross-org access without operator bypass.

**Trade-off:** Debugging is slightly harder (can't distinguish "wrong org" from "typo in org ID"), but security is meaningfully improved for multi-tenant scenarios.

## Non-blocking Audit Writes

**Decision:** Audit events are queued in an mpsc channel and written by a background worker, not written inline in the handler.

**Context:** Writing to the database adds latency to every request. Audit writes are important for compliance but are not on the critical path of the response — the caller doesn't need to wait for them.

**Choice:** `ChannelAuditRecorder::record()` sends to a bounded channel (default capacity: 4096). The `AuditWorker` drains the channel asynchronously and writes to the database with configurable retries and exponential backoff.

**Trade-off:**
- **Pro:** Audit writes add zero latency to HTTP responses. Failed DB writes are retried.
- **Con:** Under extreme load the channel can fill and events are dropped (with an error log). If 4096 events queue up without the worker draining them, something is already catastrophically wrong.
- **Con:** In tests, `BlockingAuditRecorder` must be used to make audit assertions deterministic.

See [[Audit-System]] for the full architecture.

## JWT + Revocation List (not server-side sessions)

**Decision:** Stateless JWTs with a revocation table for logout, rather than server-side session storage.

**Context:** Pure stateless JWTs can't be invalidated before they expire — logout does nothing. Full server-side sessions add a per-request DB lookup.

**Choice:** JWTs with a `revoked_tokens` table. On logout, the token's `jti` is inserted. The auth middleware checks this table on every request. The table only needs to hold entries until their `exp` passes — entries are naturally garbage-collectible.

**Trade-off:**
- **Pro:** Simple to implement, no session affinity needed, horizontally scalable.
- **Con:** One extra DB read per authenticated request. (Mitigated by an index on `jti`.)
- **Con:** If `revoked_tokens` is unavailable, logged-out tokens could temporarily be reused. Acceptable for this use case.

See [[Authentication#Middleware]] and [[Data-Model#`revoked_tokens`]].

## Opportunistic Password Rehash

**Decision:** On successful login, if the stored password hash uses outdated Argon2 parameters, rehash and update silently.

**Context:** Password hashing parameters need to be increased over time as hardware gets faster. Forcing all users to reset their passwords is poor UX. Rehashing on every login is wasteful.

**Choice:** `needs_rehash()` in [`src/security/service/password_hash.rs`](../../src/security/service/password_hash.rs) checks whether the stored hash's parameters match the current target. If not, the new hash is computed and saved after successful login.

**Trade-off:**
- **Pro:** Zero UX impact — users never notice. Parameter upgrades roll out organically.
- **Con:** Slightly higher CPU cost on the first login after a parameter change.

---

## UUIDs v7

**Decision:** All IDs are UUID v7 (time-ordered), generated by the application, never by the database.

**Context:** Standard UUID v4 is random — it causes B-tree index fragmentation at scale. Sequential integers are simple but leak information (total record counts, creation order). PostgreSQL `SERIAL` requires a round-trip to the database to get the generated ID.

**Choice:** UUID v7 encodes a millisecond timestamp in the high bits, making them roughly time-ordered. This preserves B-tree locality, avoids a DB round-trip for IDs, and enables cursor-based pagination with `(timestamp, id)` as a stable sort key.

**Trade-off:**
- **Pro:** Good index performance, no DB round-trip, sortable.
- **Con:** Slightly less random than v4 (timestamp is visible), but this is rarely a security concern for IDs.

## RBAC in the Database (not code)

**Decision:** Permissions are rows in a `permissions` table, role-permission mappings are in `role_permissions`, and user memberships are in `user_organisation_roles`. The code does not hardcode permission lists.

**Context:** Permission logic could be embedded directly in code (e.g., a `match` on role code). This is simpler initially but makes it impossible to extend roles without a deployment.

**Choice:** Permissions and role-permission mappings are data. Adding a new permission for a new use case requires a migration, not a code change. Custom roles (future: org-defined roles) would be possible without any application code change.

**Trade-off:**
- **Pro:** Flexible, auditable, data-driven.
- **Con:** Requires a DB query per request to load the permission set (one fast indexed query).
- **Con:** Slightly more migration overhead when adding a new permission.

---

## One File Per Use Case

**Decision:** Each service function lives in its own file, not in a monolithic `service.rs`.

**Context:** A `service.rs` that grows to contain all use cases for a domain becomes difficult to navigate and causes merge conflicts.

**Choice:** `src/security/service/login.rs`, `src/security/service/register_user.rs`, etc. Each file exports `Input`, `Output`, `Error`, and one function. The function is named after the use case, not the file.

**Trade-off:**
- **Pro:** Self-contained, easy to find, minimal coupling between use cases.
- **Con:** More files. Navigating requires knowing what to look for. Mitigated by the consistent naming convention.

---

## No Email Delivery

**Decision:** The password reset flow generates a token and logs it, but doesn't send email.

**Context:** Email delivery (SMTP, SES, SendGrid) adds a significant external dependency to a seed project. The exact delivery mechanism varies widely by deployment.

**Choice:** The token is logged at `INFO` level. The integration point is clearly marked — `src/security/service/password_reset_request.rs` has the `// TODO: send email` comment replaced with a clear log statement that would be swapped out in production.

**Trade-off:**
- **Pro:** No external service dependency in the seed. The flow is fully testable.
- **Con:** Not production-ready as-is for actual password reset.

---

## Cursor Encoding: base64url(JSON)

**Decision:** Cursors are `base64url(json(T))` rather than, say, encrypted or signed opaque tokens.

**Context:** Cursors need to be (1) opaque to callers, (2) decodable by the server, (3) safe in URLs.

**Choice:** Simple base64url-encoded JSON. URL-safe, no dependencies, trivially decodable by the server.

**Trade-off:**
- **Pro:** Zero crypto overhead, easy to debug.
- **Con:** A sufficiently motivated client can decode the cursor and understand its structure. But since cursors contain only timestamps and IDs — neither sensitive nor actionable — this is acceptable. The API contract says "treat as opaque", not "this is secret."
- **Con:** Not stable across version changes to the cursor struct.

See [[Pagination]].

---

## Trait Objects over Generics for AppState

**Decision:** `AppState` holds `Arc<dyn Trait>` (trait objects), not generic parameters `<R: UserRepository>`.

**Context:** Both approaches allow dependency injection and mocking. Generics are zero-cost at runtime. Trait objects have a small vtable dispatch overhead.

**Choice:** Trait objects. They keep `AppState` a concrete type that can be passed to Axum's `State<AppState>` extractor without complex generic bounds propagating through every handler.

**Trade-off:**
- **Pro:** Simple, no generic type parameters on handlers, works cleanly with Axum's `Clone` requirement.
- **Con:** Small runtime overhead from vtable dispatch. Negligible compared to the DB queries the methods execute.
